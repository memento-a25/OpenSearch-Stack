## Отчет о выполнении тестового задания:

### 1. Подготовка окружения (Prerequisites)

Стенд развернут на ВМ под управлением **AlmaLinux 9** (развернута через Cloud-Init с двумя сетевыми интерфейсами для управления и доступа к Dashboards). Перед запуском контейнеров выполнены обязательные настройки ядра ОС:

- **Оптимизация памяти:** `sysctl -w vm.max_map_count=262144` (необходимо для работы OpenSearch).
    
- **Управление ресурсами:** Отключен SWAP (`swapoff -a` + комментирование строки в `/etc/fstab`, если есть) для предоставления гарантированной памяти согласно Best-Practise.
    
* Также необходимо установить docker набор:

docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
  
  
### 2. Архитектура решения

Выбрана схема **Filebeat → Logstash → OpenSearch**, развернутая через Docker Compose.

**Ключевое проектное решение:** Изначально рассматривался вариант прямой отправки логов из Filebeat в OpenSearch. Была написана вся логика парсинга и удаление событий DEBUG через Filebeat. 

Однако, столкнувшись с ограничениями лицензионных проверок в новых версиях Filebeat (9.x+), и нежеланием переключаться на OpenSearch версий `1.x` или `2.x` для совместимости было принято решение добавить в стек **Logstash** с плагином `logstash-output-opensearch`. Это позволило:

1. Использовать актуальные версии инструментов.
    
2. Вынести тяжелую логику парсинга и фильтрации с агента сбора на промежуточный слой (Best Practice).
    
3. Гибко управлять индексацией на основе метаданных лога.

4. Поработать с важным инструментом обработки логов. 
    

### 3. Сбор и парсинг данных

Логи собираются Filebeat из статического файла (в Docker-контейнере смонтирован как volume для воспроизводимости). Далее данные передаются в Logstash, где реализован многоступенчатый пайплайн:

- **Dissect:** Первичная нарезка строки на высокоуровневые поля (`date_time`, `src`, `severity`, `msg`).
    
- **Grok:** Точечное извлечение `PID` процесса из текстового сообщения для построения аналитических таблиц.
    
- **Drop:** Фильтрация событий уровня `DEBUG` согласно ТЗ.
    
- **Динамическая индексация:** Реализовано распределение данных по индексам на основе поля хоста: `index => "%{src}-index"`.
    

### 4. Безопасность (Security)

Для демонстрации использован Demo-режим Security плагина с кастомным паролем администратора.

**Важно (Production Readiness):** В реальной среде необходимо: 

1. Заменить самоподписанные сертификаты на доверенные (TLS). 

2. Настроить RBAC (Role-Based Access Control). 

3. Установить агенты Filebeat нативно на хосты (RPM/DEB) вместо Docker-контейнеров.

4.  И многое другое...

### 5. Визуализация (Dashboards)

Подготовлен дашборд из 5 панелей, закрывающий все требования ТЗ:

1. **Круговая диаграмма:** Распределение событий по критичности.
    
2. **Таблица:** Топ-5 процессов, генерирующих ошибки (используется поле `pid`).
    
3. **Линейный график:** Динамика событий (в мс, с учетом специфики загрузки логов за 2 секунды).
    
4. **Столбчатая диаграмма:** Распределение типов ошибок в разрезе каждого хоста. Настроены через filters для статичности (INFO всегда внизу, CRITICAL всегда вверху)
    
5. **Сырые данные:** Таблица сырых логов с настроенным набором колонок для удобства расследования инцидентов.


## Post Mortem


### 1. Filebeat и ситуация с лицензиями


- **Проблема:** При попытке использовать актуальные версии Filebeat (9.x) возникла ошибка несовместимости. Filebeat пытается проверить лицензию на стороне сервера, что приводит к отказу в работе с OpenSearch.

**Варианты решения проблемы:**

*  Использовать OpenSearch более низких версий (`1.x` или `2.x`) и использовать маскировку под версию 7.10.2 Elasticsearch + использовать старую версию filebeat (`7.x - 7.12.x`)

* Использовать современные версии OpenSearch и Filebeat, но добавить в стек Logstash (с которым я до этого не работал и хотел попробовать поработать) чтобы filebeat отсылал логи на лицензированный Logstash, а на стороне Logstash установить официальный плагин logstash-output-opensearch для общения с OpenSearch без проблем с лицензированием.
  
	Я выбрал второй вариант, чтобы использовать актуальные версии, современные инструменты и не заниматься шаманством. 
	
	Логику парсинга пришлось переписывать уже на язык Logstash и учиться уже ему, по документации.


### 2. Повторный парсинг логов

- **Проблема:** В процессе добавления нового поля "PID", уже ПОСЛЕ загрузки и обработки Logstash'ем всех индексов, необходимо было ещё раз пропустить весь этот массив через filebeat (который уже запомнил в своем registry что он прочитал весь этот файл и отправил на logstash), а потом и сам Logstash, с новым правилом фильтрации, чтобы в OpenSearch БД я мог увидеть на всех документах поле PID.
  
- **Решение:**  Удален контейнер и вольюм filebeat, как и индексы в OpenSearch. **Метод грубый и его следует применять только в "песочнице" подобной моей.** 
    
- **Итог:** Filebeat перечитал и снова залил логи в Logstash, а Logstash в БД, поле PID появилось на каждом документе.

В продакшене я бы создал правило автоматической обработки (**Ingest Pipeline**) и перелил данные из старого индекса в новый прямо через него. Это позволило бы на лету добавить поле `pid` во все старые документы, не удаляя их и не останавливая работу дашбордов.
